# K230 YOLO 视觉识别方案总结

本文记录钢球识别从 FOMO 试验切换到 YOLOv8，并最终在 K230 上稳定运行约 30 FPS 的方案。目的不是只复现这一个模型，而是形成后续目标检测项目可以直接套用的开发、转换、部署和排错流程。

本文对应的已验证板端环境为 `CanMV v1.8`、`k230_canmv_v3p0`（固件构建日期 `2026-07-03`）。升级或更换固件后，应重新核对 `nncase_runtime`、AI2D 和媒体管线 API。

## 1. 最终结论

本项目最终采用以下链路：

```text
YOLOv8n 320 训练权重 (.pt)
  -> 导出静态 ONNX，输出 (1, 5, 2100)
  -> 在 ONNX 图内增加 TopK，只保留置信度最高的 32 个候选
  -> nncase PTQ 转换为 uint8 Kmodel，输出 (1, 5, 32)
  -> K230 AI2D 完成 uint8 NCHW letterbox
  -> KPU 推理
  -> MicroPython 完成阈值过滤、NMS、坐标还原和 OSD 绘制
```

当前已验证配置：

| 项目 | 配置 |
|---|---|
| 模型 | YOLOv8n，单类别 `steel_ball` |
| 模型输入 | `1 x 3 x 320 x 320`，NCHW |
| 相机 AI 通道 | `640 x 360` |
| 显示 | `virt`，`640 x 360` |
| Kmodel 输入 | `uint8` |
| Kmodel 输出 | `1 x 5 x 32` |
| 置信度阈值 | `0.20` |
| NMS IoU 阈值 | `0.45` |
| 最大输出数量 | `8` |
| 实测帧率 | 约 30 FPS |

当前模型文件为 `steelball_yolov8n_320_topk_uint8.kmodel`，板端参考程序为 `yolo.py`。

## 2. 为什么最终没有继续使用 FOMO

FOMO 更适合只需要目标中心点、目标相对较大且外观稳定的场景。钢球直径只有 1 cm，远距离时在画面中只占很少像素，而且项目还需要较准确的框和位置用于小车接近、吸附，因此 FOMO 的局限比较明显：

- FOMO 原生输出更接近热力图中心点，显示出来的框是根据网格或连通区域估算的，不是直接回归得到的边界框。
- 小球在低分辨率特征图上容易丢失；反光、阴影和地面高光也容易形成峰值。
- 多个相邻小球可能合并成一个响应区域，或者一个球产生多个峰。
- 本次 FOMO 版本曾出现约 0.2 FPS，但这不是 FOMO 必然只能达到的速度，而是当时模型接口、输出解析和板端处理链路共同造成的结果。

YOLO 的边界框回归更符合本项目需求。经过 TopK 裁剪后，既保留了 YOLO 的检测能力，也避免在 MicroPython 中每帧处理全部 2100 个候选。

## 3. 模型训练与导出原则

### 3.1 数据集比模型参数更重要

钢球属于典型的小目标和强反光目标。训练集必须覆盖实际比赛条件：

- 不同距离和像素尺寸，尤其要增加远距离小球。
- 1 到 8 个球的各种数量和分布。
- 地面颜色、纹理、接缝、灰尘和阴影。
- 顶光、侧光、曝光变化以及钢球高光位置变化。
- 小车运动造成的模糊、俯视角变化和画面边缘目标。
- 容易误检的圆形高光、螺钉、反光点和地面污渍，作为负样本保留。

标注框应贴合球的可见轮廓，并保持同一套标注标准。框偏大、偏小或漏标会直接影响小目标定位。

如果远处钢球在原始相机画面中只有几像素，单纯降低阈值无法补回信息。应优先调整镜头视场、安装高度、照明或使用 ROI，再考虑提高模型输入尺寸。

### 3.2 固定使用静态 320 输入

本次稳定模型重新按 `320 x 320` 训练和导出，部署侧也保持 320 输入。导出设置为：

```python
model.export(
    format="onnx",
    imgsz=320,
    batch=1,
    dynamic=False,
    simplify=False,
    opset=11,
    nms=False,
)
```

重要原则：训练、ONNX、nncase 和板端 AI2D 的输入尺寸及预处理必须完全一致。不要只修改板端常量而继续使用另一尺寸训练或导出的模型。

单类别 YOLOv8 的标准导出接口应为：

```text
input:  (1, 3, 320, 320)
output: (1, 5, 2100)
```

其中 5 个通道为 `cx, cy, w, h, score`，2100 来自 40x40、20x20、10x10 三个尺度。

## 4. TopK 是本次性能优化的关键

标准 YOLO 输出有 2100 个候选。KPU 虽然可以很快完成网络推理，但将全部结果交给 MicroPython 遍历、排序和 NMS 会增加 CPU、内存和数据搬运开销。

`export_yolo_topk.py` 在 ONNX 图尾部增加：

```text
Gather score plane
  -> TopK(32)
  -> Expand indices
  -> GatherElements boxes and scores
```

这样 KPU 最终只输出 `(1, 5, 32)`，候选已按分数从高到低排列。板端遇到第一个低于阈值的候选即可停止遍历，再对少量候选执行 NMS。

TopK 数量的选择原则：

- 必须高于场景真实目标上限。本项目最多 8 个球，选择 32 留出了重复框和误检余量。
- 太小可能在 NMS 前就丢掉真实目标。
- 太大则增加板端后处理开销，但不一定提高召回率。

不要把 TopK 数量和最终检测数量混为一谈。本项目是 `TopK=32`，NMS 后最多保留 `8` 个目标。

## 5. nncase 量化与转换

### 5.1 使用 uint8 PTQ

K230 部署最终选择 uint8，而不是 float32。主要原因是 uint8 更适合 KPU 的性能和内存约束。当前转换参数为：

```text
target          = k230
input_type      = uint8
input_layout    = NCHW
input_shape     = [1, 3, 320, 320]
input_range     = [0, 255]
preprocess      = True
mean            = [0, 0, 0]
std             = [255, 255, 255]
quant_type      = uint8
w_quant_type    = uint8
calibrate_method= Kld
```

校准数据使用 50 张具有代表性的真实图片。校准图不能只选最清晰、最近的钢球，应覆盖暗光、反光、远距离、无目标和多目标画面。

### 5.2 预处理必须逐项一致

PC 转换端与 K230 板端都使用：

- RGB 三通道。
- NCHW 排列。
- `uint8` 输入。
- 保持比例缩放。
- 居中 letterbox。
- 填充值 `[114, 114, 114]`。
- 双线性插值。

只要通道顺序、布局、归一化或 padding 有一项不一致，就可能出现模型能运行但置信度异常、框位置错误或完全不识别。

### 5.3 推荐转换顺序

```powershell
python export_yolo.py --imgsz 320
python export_yolo_topk.py
python convert_yolo.py --samples 50
python convert_yolo_topk.py
```

执行前需把 `convert_yolo.py` 和 `convert_yolo_topk.py` 中的校准图片目录改为当前数据集路径。

转换不能只看“生成了 kmodel”。至少要检查：

1. ONNX checker 通过。
2. 输入输出形状符合预期。
3. ONNX Runtime 能运行。
4. nncase Simulator 能运行。
5. ONNX 与 Kmodel 的框、分数误差在可接受范围内。
6. TopK 模型与标准模型的候选结果一致。

## 6. K230 板端实现要点

### 6.1 直接使用官方 PipeLine 和 AIBase 结构

相机和媒体资源初始化对顺序、通道和缓冲区很敏感。本次稳定版本沿用 CanMV `16-AI-Cube` 例程中的 `PipeLine`、`AIBase` 和 `Ai2d` 组织方式，只替换模型特有的预处理与后处理。

推荐配置：

```python
RGB888P_SIZE = [640, 360]
DISPLAY_SIZE = [640, 360]
DISPLAY_MODE = "virt"
CAMERA_FPS = 30
```

AI 通道宽度需要按 16 对齐。当前 640 已对齐，通用代码仍应保留 `align_up(width, 16)`。

### 6.2 使用 NCHW uint8

当前固件已验证的 AI2D 格式是：

```python
self.ai2d.set_ai2d_dtype(
    nn.ai2d_format.NCHW_FMT,
    nn.ai2d_format.NCHW_FMT,
    np.uint8,
    np.uint8,
)
```

不要照搬其他版本文档里的 `NHWC_FMT`，也不要假定 `nncase_runtime` 或 `ulab.numpy` 拥有桌面环境中的全部 dtype 属性。本次固件曾分别出现：

```text
ai2d_format has no attribute NHWC_FMT
module has no attribute float32
```

这类错误首先说明固件 API 与参考代码版本不一致，不代表模型损坏。

### 6.3 letterbox 坐标必须逆变换

相机画面为 16:9，模型输入为正方形。预处理时画面按比例缩放并上下填充，输出框仍处于 320x320 模型坐标中。绘制前必须执行逆变换：

```text
x_original = (x_model - pad_x) / scale
y_original = (y_model - pad_y) / scale
```

然后将坐标裁剪到相机范围，最后按显示尺寸缩放。框整体偏移、比例不对或贴不到球上时，应优先检查这里，而不是先重训模型。

### 6.4 OSD 绘制

每帧先清空 OSD，再绘制状态、框和置信度，最后调用 `show_image()`：

```text
get_frame
  -> detector.run
  -> osd_img.clear
  -> draw boxes and labels
  -> show_image
```

检测坐标属于 AI 通道，绘制坐标属于显示通道，两者分辨率不同时必须缩放。标签位置要限制在屏幕内，避免框靠近顶部时文字不可见。

### 6.5 内存和实时性

- 不要每帧创建大量长期存活的数组或张量。
- 不要每帧执行 `gc.collect()`；当前程序每 10 帧回收一次。
- 前 3 帧打印 profile，之后关闭 debug，避免串口日志拖慢循环。
- 每秒打印一次 `balls/max/fps` 足够，不要每帧打印大数组。
- `finally` 中必须依次释放 detector、pipeline 并执行 GC。

## 7. 本次典型故障及判断方法

### 7.1 `sensor snapshot chn(2) failed(3)`

常见原因：

- 上一次脚本异常退出，没有释放相机或媒体资源。
- IDE、另一个脚本或显示管线仍占用相机。
- 自己重写 Pipeline 时通道配置与官方例程不一致。
- 相机刚启动，首帧暂时未就绪。

处理顺序：

1. 先运行 `camera_pipeline_test.py`，只采集 10 帧，不加载模型。
2. 保证脚本有 `try/finally` 和 `pipeline.destroy()`。
3. 捕获首帧失败后最多重试 3 次，每次间隔 100 ms。
4. 仍失败时停止 IDE 中其他任务或重启板卡，再测试官方相机例程。

如果官方例程稳定而自己的脚本失败，优先判断为资源生命周期或通道配置问题，不要先怀疑摄像头损坏。

### 7.2 `KPU run failed`

这个错误信息本身不足以定位。应拆成以下步骤测试：

```text
加载 kmodel
  -> 检查 inputs_size/outputs_size
  -> 构造固定 uint8 输入
  -> 单独 kpu.run
  -> 读取每个输出及 shape
  -> 再接 AI2D
  -> 再接相机
  -> 最后接后处理和 OSD
```

当前 `yolo_inference_test.py` 就是最小 KPU 隔离测试。它使用常量 `(1,3,320,320)` uint8 输入，并要求输出为 `(1,5,32)`。

若最小测试失败，应检查模型文件、输入类型、输入形状和转换版本；若最小测试通过而完整程序失败，应检查 AI2D、内存和资源初始化顺序。

### 7.3 加载或运行后 IDE 断开

本次 GFL 三输出模型分别产生：

```text
(1, 65, 40, 40)
(1, 65, 20, 20)
(1, 65, 10, 10)
```

模型本身可以单独运行，但在当前 CanMV 固件中，GFL 模型、AI2D 与 `aicube` 的组合出现原生内存冲突，表现为初始化后断开，而不是正常抛出 Python 异常。

经验：

- “能加载模型”不等于“完整管线可用”。
- 设备断开通常要怀疑 native 崩溃、内存冲突或供电，不应直接认定芯片损坏。
- 本项目最终绕开 `aicube` 和三输出 GFL，使用单输出 TopK 模型加轻量 Python NMS，稳定性明显更好。

### 7.4 模型能运行但结果一直不变

要同时判断输入是否在变化和输出是否合理：

- 打印低成本的帧签名或少量像素统计，不要只看 `balls/max`。
- 记录输出 shape、最高分数和最高分索引。
- 移动物体后确认帧签名、峰值位置和框坐标都发生变化。

如果输入不变，排查相机取帧；如果输入变化而输出不变，排查预处理、量化和模型；如果数值变化但框不动，排查输出维度解析和坐标变换。

### 7.5 `Kmodel file not exist`

先在板端确认绝对路径。Windows 资源管理器可能隐藏 `.py` 和 `.kmodel` 扩展名，但程序中必须使用完整路径：

```python
KMODEL_PATH = "/sdcard/app/steelball_yolov8n_320_topk_uint8.kmodel"
```

模型更新后要同步修改文件名，避免板上存在多个近似名称导致误用旧模型。

## 8. 推荐的分层验证流程

以后部署任何新视觉模型，都按以下顺序推进，不要直接运行完整小车程序：

1. **PC 模型验证**：用训练框架对真实图片推理，确认模型精度。
2. **ONNX 验证**：检查输入输出 shape，并与原模型结果对比。
3. **Kmodel 模拟器验证**：确认量化误差和输出维度。
4. **板端 KPU 隔离测试**：使用固定输入，只加载和运行模型。
5. **相机隔离测试**：不加载模型，连续取帧并正确释放资源。
6. **AI2D + KPU 测试**：不绘制、不控制电机，先看输出统计。
7. **OSD 测试**：验证坐标逆变换、显示缩放和多个框。
8. **长时间稳定性测试**：至少连续运行 10 到 30 分钟，观察 FPS、内存和断连。
9. **接入运动控制**：最后再把检测结果交给小车或云台。

每一层通过后再增加下一层。这样可以明确问题属于模型、转换、相机、运行时、绘制还是控制逻辑。

## 9. 远距离小球的优化方向

远距离识别差首先是成像问题，其次才是阈值问题。建议按以下优先级优化：

1. 统计比赛最远距离下钢球在原图中的直径像素数。
2. 增加真实安装高度、真实地面和真实光照下的远距离训练样本。
3. 收紧摄像头视场或使用更合适焦距，让目标占更多像素。
4. 固定曝光、增益和照明，减少钢球高光漂移与运动模糊。
5. 若循迹已经给出小球可能区域，只裁剪 ROI 推理，提高有效像素密度。
6. 必要时再试 416 或 640 输入，并重新测 FPS、内存和稳定性。

不建议仅靠把置信度从 0.20 一直降低来解决远距离漏检。阈值过低会把地面反光和圆形纹理一起放大。

## 10. 接入小车控制时的输出设计

识别层不应直接发送“左转/右转”，而应输出稳定、可解释的目标数据，例如：

```python
{
    "found": True,
    "count": 3,
    "target": {
        "cx": 412,
        "cy": 285,
        "w": 24,
        "h": 23,
        "score": 0.87,
    },
}
```

控制层再根据任务选择目标：

- 吸附任务通常优先选择画面下方、离小车最近且置信度稳定的球。
- 搜索阶段可选择最靠近画面中心或规划方向的球。
- 连续 2 到 3 帧出现后再确认目标，减少闪框触发误动作。
- 短暂丢失时保持上一目标少量帧，超过超时立即停车或进入搜索。
- 用中心点滑动平均或指数滤波，避免检测框抖动直接传给转向控制。
- 检测循环可保持 30 FPS，但控制命令无需每帧发送；限制命令频率并设置死区。

视觉坐标到小车运动命令之间必须有明确的坐标约定、限幅、超时和失效停车逻辑。

## 13. 最值得复用的经验

1. **先确认接口，再调识别效果。** 输入类型和输出 shape 不明确时，画框与阈值调试没有意义。
2. **先做隔离测试，再做完整集成。** 相机、KPU、AI2D、后处理和显示应分别证明可用。
3. **PC 能跑不代表 K230 能稳定跑。** 还要验证 nncase、固件 API、native 内存和媒体资源组合。
4. **小目标首先解决像素和数据问题。** 阈值只能改变取舍，不能创造图像细节。
5. **把重计算放进模型图或 KPU。** TopK 将 2100 个候选缩减到 32，是本次恢复实时性的关键。
6. **板端后处理保持简单、固定上限。** 少量候选、提前停止、轻量 NMS 和低频日志更适合 MicroPython。
7. **坐标系统必须成对验证。** 模型框正确但 OSD 不准，通常是 letterbox 或显示映射错误。
8. **断连不等于硬件损坏。** 先用最小用例定位 native 冲突、内存和资源生命周期。
9. **模型文件保持唯一和可追溯。** 文件名、哈希、输入输出接口和阈值应随版本一起记录。
